Feature Detection & Matching
SIFT, Descriptors & the Ratio Test

SfM starts by finding pixels in two photos that show the same physical point. This chapter covers how SIFT detects and describes those pixels, and how the ratio test filters the resulting matches. Source: features.py.

The Pipeline at a Glance

casualSfM is a two-view reconstruction pipeline with four stages. This chapter is stage 1.

images
  → [Ch 01] feature detection & matching   ← you are here
  → [Ch 02] fundamental matrix + RANSAC
  → [Ch 03] pose recovery + triangulation
  → [Ch 04] evaluation + visualisation

All stages are independent modules. The output of stage 1 is two NumPy arrays of pixel coordinates (one per image) of shape (K, 2) float32.

SIFT Detection: Finding Keypoints

SIFT (Lowe, 2004) finds blob-like structures by looking for extrema in the Difference of Gaussians (DoG) scale space. At each scale σ:

Math

D(x, y, σ) = ( G(x, y, kσ) − G(x, y, σ) ) ∗ I(x, y)

G(x, y, σ) = (1/2πσ²) · exp(−(x²+y²)/2σ²)

Keypoint candidate: local extremum of D
in a 3×3×3 neighbourhood (x, y, scale)

Code

sift = cv.SIFT_create()

kp1, desc1 = sift.detectAndCompute(img1, None)
kp2, desc2 = sift.detectAndCompute(img2, None)

# kp:   list of cv.KeyPoint objects
#        .pt    → (x, y) pixel position
#        .size  → detected scale
#        .angle → dominant orientation
# desc: float32 array, shape (N, 128)

Keypoints are refined to sub-pixel precision by fitting a 3D quadratic to the DoG extremum. Weak-contrast candidates are discarded via a contrast threshold; edge-like responses are discarded via the ratio of principal curvatures.

SIFT Descriptor: 128-D Fingerprint

Around each keypoint a 16×16 patch is extracted, aligned to the keypoint's dominant gradient orientation. The patch is divided into a 4×4 grid of sub-regions; each sub-region contributes an 8-bin gradient histogram.

Math

Patch: 16×16 px, rotated to keypoint orientation
Grid:  4×4 cells, each 4×4 px
Bins:  8 gradient orientations per cell

Descriptor: 4 × 4 × 8 = 128 floats

Post-processing:
  d ← d / ‖d‖₂           (unit norm)
  d ← clip(d, 0, 0.2)    (clamp large values)
  d ← d / ‖d‖₂           (renormalise)

Code

# detectAndCompute handles all of this:
# detection + orientation + descriptor
# in a single OpenCV call.

# desc1.shape → (N, 128)
# dtype       → float32
# norm        → ≈ 1.0 (unit length)

# The 128-D vector encodes gradient
# structure, not raw pixel intensities,
# making it robust to lighting change.

Brute-Force Nearest Neighbour

For each descriptor in image 1, find the two closest descriptors in image 2 by Euclidean distance. With N ≈ 8,000 keypoints per image this is ~8k × 8k × 128 multiplications, feasible via BLAS in OpenCV's BFMatcher.

Math

d(a, b) = ‖a − b‖₂ = √Σᵢ(aᵢ − bᵢ)²

For each desc in image 1:
  m = argmin   d(desc, desc2[j])
       j
  n = argmin   d(desc, desc2[j])
       j ≠ m
# m → best match, n → second-best

Code

bf = cv.BFMatcher()   # L2 norm (for SIFT floats)
                      # use NORM_HAMMING for ORB

raw = bf.knnMatch(desc1, desc2, k=2)
# raw: list of (DMatch m, DMatch n)
# m.distance → d(best)
# n.distance → d(second-best)

Lowe's Ratio Test

A match is ambiguous if the second-best descriptor is nearly as close as the best. The ratio test discards those ambiguous cases.

Math

Keep match m iff:

  d(m) / d(n) < τ,   τ = 0.75

Interpretation:
  If best match is clearly closer than
  second-best → distinctive → keep.
  If they are nearly equal → ambiguous
  (repetitive texture) → discard.

Code

good = [
    m for m, n in raw
    if m.distance < 0.75 * n.distance
]

# Extract pixel coordinates
pts1 = np.float32(
    [kp1[m.queryIdx].pt for m in good])
pts2 = np.float32(
    [kp2[m.trainIdx].pt for m in good])

# Output: float32 arrays, shape (K, 2)
Key insight The ratio test is scale-free. It doesn't matter how large the distances are in absolute terms, only their relative separation. Lowe reported 90% of false matches eliminated at τ = 0.8 with 95% of correct matches retained.

The Full Class

class FeatureMatcher:
    def __init__(self, ratio_threshold: float = 0.75):
        self.sift = cv.SIFT_create()
        self.bf   = cv.BFMatcher()        # L2 norm by default
        self.ratio_threshold = ratio_threshold

    def __call__(self, img1_path, img2_path):
        img1 = cv.imread(img1_path, cv.IMREAD_GRAYSCALE)
        img2 = cv.imread(img2_path, cv.IMREAD_GRAYSCALE)

        kp1, desc1 = self.sift.detectAndCompute(img1, None)
        kp2, desc2 = self.sift.detectAndCompute(img2, None)

        raw  = self.bf.knnMatch(desc1, desc2, k=2)
        good = [m for m, n in raw
                if m.distance < self.ratio_threshold * n.distance]

        pts1 = np.float32([kp1[m.queryIdx].pt for m in good])
        pts2 = np.float32([kp2[m.trainIdx].pt for m in good])
        return pts1, pts2
Numbers on our fountain sequence SIFT detects ~8,000 keypoints per image. knnMatch returns ~4,000 raw pairs. After the ratio test at τ = 0.75, roughly 1,500–2,000 survive. After RANSAC in Ch 02, ~900 are epipolar inliers. After cheirality in Ch 03, the final count is 701 triangulated points.

Pitfalls

Load in grayscale. Passing a 3-channel BGR image to detectAndCompute causes an internal format error. Always use cv.IMREAD_GRAYSCALE.

L2 vs Hamming. BFMatcher() defaults to L2, correct for SIFT's float descriptors. ORB produces binary descriptors, so use BFMatcher(cv.NORM_HAMMING) instead. Mixing them gives nonsensical distances.

Ratio threshold is scene-dependent. 0.75 works well on our richly textured fountain. On repetitive scenes (uniform brickwork, tiled floors) lower τ to 0.6 to avoid ambiguous matches, accepting fewer total correspondences.

These matches still contain outliers. The ratio test is permissive by design. RANSAC in Ch 02 is what actually rejects wrong correspondences.
