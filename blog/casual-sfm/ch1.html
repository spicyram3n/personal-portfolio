Feature Detection & Matching

SIFT keypoints, the 128-dimensional descriptor, and Lowe's ratio test.

Structure from Motion begins with a simple question: which pixels in two photographs show the same point in the world? This chapter answers it. We find distinctive points with SIFT, describe each one as a vector, match those vectors between images, and throw away the matches we can't trust. Source: features.py.

The pipeline

casualSfM turns two photographs into a small 3D point cloud. It runs in four stages, and each stage is an independent module. This chapter is stage one.

The job of stage one is to produce two arrays of pixel coordinates, one per image, of shape (K, 2). Row k in the first array and row k in the second point to the same physical feature, seen from two camera positions. Everything downstream is built on these correspondences.

Finding keypoints

We need points the algorithm can find again in a second photo, even when the camera has moved and the lighting has shifted. Corners and blobs work well for this. Flat walls and straight edges do not, because they look the same as you slide along them.

SIFT (Lowe, 2004) finds blob-like points by building a scale space: it blurs the image with Gaussians of increasing width and subtracts neighbouring blur levels. This difference-of-Gaussians response peaks wherever a blob sits at a matching size, so a peak locates a feature in both position and scale.

The math

A Gaussian blur at width \sigma:

G(x,y,\sigma) = \frac{1}{2\pi\sigma^2}\, \exp\!\left(-\frac{x^2+y^2}{2\sigma^2}\right)

Subtract two nearby blur levels to get the response we search:

D(x,y,\sigma) = \big(G(x,y,k\sigma) - G(x,y,\sigma)\big) * I(x,y)

A keypoint is a local maximum or minimum of D across a 3\times3\times3 neighbourhood in space and scale.

The code

sift = cv.SIFT_create()

kp1, desc1 = sift.detectAndCompute(img1, None)
kp2, desc2 = sift.detectAndCompute(img2, None)

# kp:   list of cv.KeyPoint
#   .pt    -> (x, y) pixel location
#   .size  -> detected scale
#   .angle -> dominant orientation
# desc: float32 array, shape (N, 128)

OpenCV refines each peak to sub-pixel accuracy, then discards two kinds of weak detections: low-contrast points that are probably noise, and points that lie along an edge rather than on a true corner.

The descriptor

A location alone isn't enough to match a point. We also need a fingerprint: a compact summary of the pixels around the keypoint that stays stable when the image is rotated or relit.

SIFT builds this from gradients. It takes a 16×16 patch around the keypoint, rotates the patch to the keypoint's dominant gradient direction (which is what buys rotation invariance), splits it into a 4×4 grid of cells, and builds an 8-bin histogram of gradient directions in each cell.

The math

Stacking every cell histogram gives the descriptor length:

\underbrace{4 \times 4}_{\text{cells}} \times \underbrace{8}_{\text{bins}} = 128

It is then normalised so brightness changes don't matter:

\mathbf{d} \leftarrow \frac{\mathbf{d}}{\lVert \mathbf{d} \rVert_2}, \quad \mathbf{d} \leftarrow \min(\mathbf{d},\,0.2), \quad \mathbf{d} \leftarrow \frac{\mathbf{d}}{\lVert \mathbf{d} \rVert_2}

Clamping at 0.2 stops any single strong gradient from dominating the vector.

The code

# detectAndCompute already did this:
# detection, orientation, and the
# descriptor in one OpenCV call.

desc1.shape   # -> (N, 128)
desc1.dtype   # -> float32
# each row has unit length (~1.0)

# The vector encodes gradient
# structure, not raw intensities,
# so it survives lighting changes.

The result is one 128-number vector per keypoint. Two photos of the same scene produce two clouds of these vectors, and matching becomes a nearest-neighbour search between them.

Matching

For each descriptor in the first image, we find its closest descriptor in the second by ordinary Euclidean distance. We actually keep the two closest, because the runner-up is what tells us whether the best match can be trusted.

The math

Distance between two descriptors:

d(\mathbf{a}, \mathbf{b}) = \lVert \mathbf{a} - \mathbf{b} \rVert_2 = \sqrt{\sum_{i=1}^{128} (a_i - b_i)^2}

For each descriptor in image 1, take the two nearest in image 2:

m = \arg\min_{j} d(\mathbf{a}, \mathbf{b}_j), \qquad n = \arg\min_{j \neq m} d(\mathbf{a}, \mathbf{b}_j)

m is the best match, n the second best.

The code

bf = cv.BFMatcher()   # L2 norm, for
                      # SIFT's float vectors

raw = bf.knnMatch(desc1, desc2, k=2)
# raw: list of (m, n) pairs
#   m.distance -> nearest
#   n.distance -> second nearest

# ~8000 keypoints per image is
# feasible: the heavy lifting runs
# in optimised BLAS under the hood.

The ratio test

Here's the key idea of the whole chapter. A match is trustworthy when the best candidate is clearly closer than the second best. If the two nearest descriptors are almost the same distance away, the point is ambiguous (think repeated brickwork or tiled floors) and we drop it.

The math

Keep a match only when:

\frac{d(m)}{d(n)} < \tau, \qquad \tau = 0.75

Small ratio means the best match stands out and we trust it. Ratio near 1 means best and second-best are tied, so we discard it.

The code

good = [
    m for m, n in raw
    if m.distance < 0.75 * n.distance
]

# pull out the surviving coordinates
pts1 = np.float32(
    [kp1[m.queryIdx].pt for m in good])
pts2 = np.float32(
    [kp2[m.trainIdx].pt for m in good])
# -> two (K, 2) float arrays
Why it works The ratio test is scale-free: it cares only about the relative gap between best and second-best, never the absolute distances. Lowe reported it removes about 90% of false matches while keeping 95% of correct ones.

Full implementation

The whole stage is small enough to read in one screen.

Numbers from the fountain sequence SIFT finds about 8,000 keypoints per image. Matching returns roughly 4,000 raw pairs. The ratio test at \tau = 0.75 leaves 1,500 to 2,000. RANSAC in Ch 02 cuts that to about 900 geometric inliers, and the cheirality check in Ch 03 lands on a final 701 triangulated points.

Pitfalls

Load images in grayscale. Passing a 3-channel colour image to detectAndCompute triggers an internal format error. Always read with cv.IMREAD_GRAYSCALE.

Match the norm to the descriptor. SIFT produces float vectors, so the L2 norm is correct. ORB produces binary vectors and needs BFMatcher(cv.NORM_HAMMING). Mixing them gives meaningless distances.

The threshold depends on the scene. 0.75 suits our richly textured fountain. On repetitive surfaces, lower it to around 0.6 to reject ambiguous matches, at the cost of keeping fewer overall.

These matches still contain outliers. The ratio test is deliberately permissive. Rejecting genuinely wrong correspondences is RANSAC's job, in the next chapter.
